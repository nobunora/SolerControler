# Naming

## Core Rule

Names should reveal the domain meaning, state, or purpose.

## Good Names

- specific nouns
- explicit verbs
- names that include units or scope when relevant
- names that match existing project vocabulary

## Avoid by Default

- `data`
- `result`
- `item`
- `value`
- `temp`
- `obj`
- `info`
- `payload`
- `params`
- `config`
- `handler`
- `manager`
- `processor`
- `service`
- `helper`
- `util`
- `process`
- `execute`
- `handle`
- `run`
- `update`
- `fix`
- `check`
- `validate`

## Stability Rule

- Do not rename public APIs, database fields, environment keys, or integration fields only for style.
- Do not rename widely used internal names unless the rename is part of the requested change.
- If a name is already a contract, treat it as a contract.
