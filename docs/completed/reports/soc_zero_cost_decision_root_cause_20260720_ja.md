# SOC 0%コスト判断の原因・再発防止・修正結果

作成日時: 2026-07-20 JST

## 作業前確認

- 変更目的: 4時計画がSOC 0%を選んだ経緯を特定し、同種の金融入力欠落と暫定契約設定の残留を再発させない。
- 変更対象: エネルギーモデルCSV入力、売電契約と目的関数の整合、本番設定、関連テスト、作業規約。
- 対象外: 消費・PV予測アルゴリズム、料金単価、蓄電池容量、KP-NET設定画面の操作手順。
- 確認した参照: Git履歴、2026-06-21の売電ペナルティ・段階料金・月次着地シミュレーション、2026-07-20保存計画、同日のKP-NET実績CSV。
- 既存設計との関係: 既存の段階料金・60シナリオSOC最適化は維持し、入力結線と契約前提を訂正した。
- 確認済み契約条件: 売電契約は未開始、売電収入は0円。

## 結論

SOC 0%は、予測された消費が少なかったためではない。次の2条件が同時に発生し、夜間充電を不当に高く、日中買電を不当に安く評価した結果である。

1. 本番CSV読込が買電・売電列を行データへ保持しておらず、7月15日から19日の日中買電66.472kWhを0kWhとして段階料金へ渡した。
2. 売電契約開始までの一時設定として導入された売電ペナルティ39.1円/kWhが残り、売電収入0円という事実を「売電すると39.1円の追加費用」と誤って評価した。

保存計画では、基準計算と類似日フィードバックは100%を支持していたが、上記のコスト目的関数が最終値を0%へ上書きした。

## なぜ既存テストを通過したか

- 月間買電テストは`{"buy": 1.0}`のような解析済み辞書を直接作り、実CSV読込を通していなかった。
- 売電モードテストは`penalty`、`neutral`、`revenue`それぞれの数式だけを確認し、契約状態との整合を検査していなかった。
- 本番デプロイスクリプトのテストは認証・バックアップ・Schedulerを検査したが、金融設定の組合せを検査していなかった。
- シミュレーションで繰り返し0kWhが出た際の停止条件がなく、異常値を入力事実として受理した。

## 修正内容

### CSV入力

- 本番CSV読込が`買電電力量[kWh]`と`売電電力量[kWh]`を保持するよう修正した。
- 買電・売電列のどちらかが欠けているCSVは、0として処理せず計画生成を停止する。

### 売電契約と費用評価

- `SOC_EXPORT_CONTRACT_STATUS`を必須化した。
- 許可する組合せを次に限定した。
  - `active + revenue + 正の売電単価`
  - `inactive + penalty/neutral`
  - `unknown + neutral`
- 確認済みの現状に合わせ、本番を`inactive + neutral + 0円`へ変更した。
- 売電未契約を理由に39.1円/kWhの架空費用を加える設定を本番から外した。

### 安全下限

- 計画SOC下限30%と4時実行時下限30%を維持した。
- `.env.example`に残っていた0%を30%へ揃えた。

## 再発防止ルール

`docs/current/agent/agent_working_rules.md`へ次を追加した。

- 金融判断に使うCSV項目は、実CSV読込から下流計算までのE2E fixtureテストを必須とする。
- 元データが非ゼロなのに集計が0になる場合は、0実績として続行せずfail-closedまたは入力不明として扱う。
- 本番化前に、少なくとも1件の生データ集計と計画記録値を照合する。
- 暫定的な料金・契約前提は本番設定で明示し、関連デプロイごとに再確認する。
- 買電は費用を下げない、認識済み売電収入は費用を上げない、契約不明をペナルティで代用しない、という方向不変条件をテストする。
- 不自然な境界値が繰り返し出た場合、感度試験で済ませず入力パイプラインを調査する。
