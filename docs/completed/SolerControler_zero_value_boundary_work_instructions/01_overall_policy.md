# SolerControler ゼロ値境界不具合 全体修正方針

> 対象: `C:\VSC\SolerControler`
> 基準HEAD: `95eb45c`
> 調査日: `2026-07-19`
> 原則: 再現済みの問題を1件ずつ修正し、既存仕様・文書・テストを削除または置換せず、必要な契約と回帰テストを追記する。

## 1. 文書の目的

この文書は、以下の4件を個別の偶発バグとして処理せず、共通原因である「数値0と欠損値の意味境界がコード上で表現されていないこと」を是正するための全体方針を定める。

1. PV総予測 `0 kWh` が欠損扱いされる。
2. SOC上限 `0%` が `100%` に変換される。
3. 気温 `0℃` が `20℃` に変換される。
4. 夜間充電開始時刻設定の名称と実際の利用契約が一致していない。

P1の2件は最終制御値を反転させるため、構造改善より先に修正する。P2の2件は、P1修正後に契約を固定してから対応する。

- フォールバック時に入力の出所と理由が失われていないか。

- 既存テストが0境界を保護しているか。
- 例外処理がフェイルセーフを維持しているか。

## 3. 調査結果の要約

### 3.1 確定した共通原因

Pythonの真偽値評価を、値の有無判定として使用している。

危険な形式:

```python
parsed_value = parse(...)
value = parsed_value or default_value
```

この形式では次を区別できない。

- `None`: 値がない。
- `0.0`: 値があり、その値が0である。
- `False`: 明示的に無効である。
- `""`: 空文字である。
- 解析失敗: 入力形式が不正である。

今回の対象では、`0.0` はすべて有効なドメイン値である。

- PV発電量0 kWh
- SOC上限0%
- 気温0℃
- 朝PV発電量0 kWh
- 昼間余剰0 kWh

### 3.2 既存テストの状態

実行結果:

```text
tests/test_energy_model.py: 37 passed
tests/test_kpnet_workflow.py: 37 passed
security_check: passed
```

既存テストは正常系と正の数値を保護しているが、以下の境界を直接検証していない。

- `predicted_pv_kwh_override=0`
- `total_kwh=0.0`
- `cap_target_soc_percent=0.0`
- `temp_c=0.0`
- `None` と `0.0` の出力差
- `KP_NIGHT_CHARGE_WINDOW_START` を既定値以外へ変更した場合の時刻生成

テスト成功は現行コードの回帰を示すが、今回の契約が正しいことは示していない。

## 4. 値の契約

修正後は、入力値を最低でも次の4分類で扱う。

| 状態 | 意味 | 原則 |
|---|---|---|
| `None` | 値が存在しない | 明示的フォールバックを許可 |
| `0.0` | 有効なゼロ値 | そのまま採用 |
| 負値 | ドメイン上不正または特殊値 | 項目別に拒否・エラー・クランプ方針を定義 |
| 解析失敗 | 文字列や型が不正 | 理由を残し、欠損とは区別 |

### 4.1 PV予測値

- `None`: 予測出力が存在しない、または予測機能が利用不能。
- `0.0`: 発電量ゼロという有効な予測。
- `< 0`: 発電量として不正。入力生成側または境界で拒否する。
- フォールバックを使用した場合は、出所と理由を出力へ残す。

### 4.2 SOC制約値

- `None`: 当該ガードが上限制約を提示していない。
- `0.0`: 上限0%という最も強い有効制約。
- `100.0`: 上限制約なしと同等の最大値。
- `< 0` または `> 100`: ガード生成側の不具合として扱い、暗黙補正しない。

### 4.3 気温

- `None`: 予報気温がない。
- `0.0`: 有効な0℃。
- 負値: 有効な気温。
- 非数値: 解析失敗として明示的既定値を使用し、理由を保持する。

### 4.4 時刻設定

- 設定名は、その値がどの計算または制御境界へ適用されるかを表す。
- 実績抽出窓と機器制御許可窓を同一設定で兼用しない。
- KP-NET機器の0時跨ぎ制約がある場合は、設定モデル、文書、テストへ明記する。

## 5. 全体修正方針

### 5.1 最小変更を優先する

最初の修正では、既存の関数構造と出力スキーマを維持する。

禁止:

- `energy_model_main.py` の全面書換え。
- 同時に複数の値正規化ロジックを移動する。
- 出力JSONのキーを変更する。
- PV予測方式そのものを変更する。
- KP-NET制御時間を仕様確認なしに変更する。

### 5.2 `is None` を使用する

既定値適用は次の形式に統一する。

```python
parsed = _to_optional_float(raw)
value = default if parsed is None else parsed
```

上限制約の集約は次の形式にする。

```python
cap = _to_optional_float(raw_cap)
if cap is not None:
    max_target_soc = min(max_target_soc, cap)
```

### 5.3 型と出所を保持する

中期的には、裸の `float | None` だけでなく、値の出所とフォールバック理由を保持する。

推奨例:

```python
@dataclass(frozen=True)
class NumericReading:
    value: float | None
    source: str
    fallback_reason: str | None
```

ただし、P1修正と同じコミットで広範囲に導入しない。まず回帰テストと最小修正を行い、その後の構造改善として実施する。

### 5.4 正規化責務を境界へ集約する

現在は `energy_model_main.py` のワークフロー内で、取得、解析、既定値適用、フォールバック選択が混在している。

目標:

- 外部取得層: 生データと取得状態を返す。
- 解析層: 型変換結果と解析失敗を返す。
- ドメイン層: 0、負値、欠損の意味を判断する。
- 最適化層: すでに契約が確定した値だけを受け取る。
- 出力層: 採用値、出所、フォールバック理由を表示する。

この分離はP1/P2修正後に段階的に行う。

## 6. 個別実施順序

### Step 1: PVゼロ予測

対象文書: `02_pv_zero_forecast.md`

理由:

- 夜間充電必要量を `6.67 kWh` から `0 kWh` へ反転させる再現がある。
- 日中買電、SOC不足、運用費増加へ直結する。
- 現行製品文書の `predicted_pv_kwh = total_kwh` 契約と矛盾する。

### Step 2: SOC上限ゼロ

対象文書: `03_soc_cap_zero.md`

理由:

- ガード上限0%を100%へ変換する。
- PV受入余地確保のための制約を完全に無効化する。
- 同型の式が複数経路に存在する。

### Step 3: 気温ゼロ

対象文書: `04_temperature_zero.md`

理由:

- 0℃を20℃へ変更し、予測PVと必要充電量を変える。
- 負温度は保持される一方、ちょうど0℃だけが誤る不連続な挙動である。

### Step 4: 夜間充電窓契約

対象文書: `05_night_charge_window_contract.md`

理由:

- 仕様判断を伴う。
- 無断変更すると実機の時刻制約を破る可能性がある。
- 先に現行挙動と設定の意味を文書・テストで固定する必要がある。

## 7. テスト全体方針

### 7.1 境界値テスト

各数値項目について最低限次を試験する。

- `None`
- `0`
- 正の最小値
- 通常値
- 上限値
- 負値
- 非数値
- 欠損とゼロで結果が異なること

### 7.2 テスト配置

- 純粋計算: `tests/test_energy_model.py`
- ワークフロー入力変換: `tests/test_energy_model_runtime.py`
- PV集計出力: `tests/test_pv_array_forecast.py`
- SOCコスト最適化制約: `tests/test_soc_cost_optimizer.py`
- KP-NET時刻生成: `tests/test_kpnet_workflow.py`

### 7.3 テストの順序

1. 現行コードで失敗する回帰テストを追加。
2. 対象問題だけを修正。
3. 対象テストを実行。
4. 関連テストを実行。
5. 全非externalテストを実行。
6. セキュリティ検査と構文検査を実行。
