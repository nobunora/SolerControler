## 1. 目的

`cloud_job_runner.py` の強制充電監視を、暗黙的なif/loop中心の処理から、明示的な状態・入力・遷移・副作用へ分離する。実機制御を含むため、可読性だけでなくfail-safe、再実行、監査可能性を強化する。

## 2. 調査済み状態

- `_monitor_partial_forced_and_stop()` は約223行。
- 初期policy判断、standby/forced設定、CSV取得、SOC抽出、cutoff、polling、failure、DB保存、停止処理を一つの関数で扱う。
- `_run_adjust_03()` は初期CSV、前日SOC feedback、night plan確保、DB pipeline、監視、export/backupを順に実行する。
- `_ensure_night_plan_available()` は既定で当日planを再生成し、Firestore fallbackも持つ。
- `_refresh_plan_for_same_date_if_changed()` は実運用から未接続で、テストからのみ参照される。
- SOC fallbackはfiniteかつ0〜100を検証する修正が入っている。
- initial SOC unavailable時は既定standbyとし、明示envでoverrideできる。

## 3. 追加調査が必要な箇所

1. KP-NETの実機状態値と書込確認方法。
2. forced書込成功後、confirm失敗・logout失敗時の実状態。

9. 操作・停止理由の監査ログ保存先。

## 4. 理想状態と現在のギャップ

| 観点 | 現在 | 理想 | ギャップ |
|---|---|---|---|
| 状態 | ローカル変数と分岐 | Enum + state data | 遷移が一覧化されない |
| 判断 | I/Oの途中で実行 | pure transition function | 単体テスト困難 |
| 副作用 | 関数内に散在 | command/effect executor | 停止漏れリスク |
| 再実行 | 暗黙 | resume/reconcile方針 | forced状態復旧が不明 |
| 監査 | print中心 | transition log + reason | 後追い分析が弱い |
| fail-safe | 複数分岐 | finally/terminal effectで保証 | 全経路保証が見えにくい |

## 5. 実装案3案

### 案A: 現行関数を小関数へ分割

初期化、poll、停止などに分けるが、状態はif/whileのまま。

- 長所: 変更が比較的小さい。
- 短所: 遷移網羅性と再実行設計が弱い。

### 案B: pure transition function + effect executor

状態、観測、policyから次状態とeffectを返す。I/Oはexecutorが実行する。

- 長所: 全遷移をテーブルテスト可能。過度なクラス化を避けられる。fail-safeを集中できる。
- 短所: state/effect model設計が必要。

### 案C: workflow engine・外部orchestrator

Temporal等のworkflow engineまたはクラウド状態機械へ移す。

- 長所: retry、resume、可観測性が強い。
- 短所:導入コストと運用複雑性が大きい。実機制御に対して過剰。

## 6. 採用案

**案Bを採用する。** 将来、jobを外部workflowへ移す場合も、pure transition modelはそのまま再利用できる。

## 8. 遷移ルール

主要ルールを表で固定する。

| 現状態 | 条件 | 次状態 | 必須effect |
|---|---|---|---|
| INITIALIZING | planなし | FAILED_COMMAND | 実機操作なし、記録 |
| INITIALIZING | SOC不明・overrideなし | HOLDING_STANDBY | SetStandby |
| INITIALIZING | SOC >= target | COMPLETED_NO_CHARGE | SetStandby, Record |
| INITIALIZING | SOC < target | STARTING_FORCED | SetForcedCharge |
| STARTING_FORCED | confirm成功 | MONITORING | FetchCsv |
| STARTING_FORCED | confirm失敗 | FAILED_COMMAND | SetStandby |
| MONITORING | SOC >= target-hysteresis | STOPPING | SetStandby |
| MONITORING | now >= cutoff | STOPPING | SetStandby |
| MONITORING | sensor failure上限 | STOPPING | SetStandby |
| MONITORING | runtime上限 | STOPPING | SetStandby |
| MONITORING | 継続 | MONITORING | Persist, Sleep, Fetch |
| STOPPING | standby確認 | terminal | Record |
| STOPPING | standby確認失敗 | FAILED_COMMAND | retry/alert |

## 13. fail-safe要件

- forcedを開始したrunは、例外時も停止処理を試行する。
- 停止失敗は成功扱いにしない。
- sensor failure時は既定standby。
- plan値はfinite、範囲、dateを検証する。
- cutoffはtimezone-awareで比較する。
- 例外を握り潰さず、terminal reasonを保存する。
- `finally` にすべてを詰めず、停止effectの結果を明示する。
