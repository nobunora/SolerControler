## 1. 目的

この段階の目的は、後続リファクタリングで「見た目は整理されたが計算結果が変わった」「backendによって結果がずれた」「停止条件が欠落した」という事態を防ぐことである。新機能は追加せず、現行挙動を実行可能な仕様として固定する。

## 2. 調査済み状態

- Pythonテストは214件通過している。
- JavaScriptの純粋計算テストはNodeで通過している。
- dashboard bootstrapにplaceholder非残存、payload注入、外部JSより前の注入、Node実行のテストがある。

## 5. 実装案3案

### 案A: 出力スナップショット中心

代表入力から生成されるJSON、DB行、dashboard payloadを丸ごと保存し、差分があれば失敗させる。

- 長所: 実装が早い。大きな差分を検知しやすい。
- 短所: 正当な追加項目にも弱い。差分原因が分かりにくい。時刻や順序で不安定になりやすい。

### 案B: 契約テスト + parity fixture + 限定snapshot

domain上重要な値は明示assert、複雑なpayloadは安定化してsnapshot、backendは同一fixtureで比較する。実機I/Oはfake/recordingで切り離す。

- 長所: 変更理由を把握しやすく、後続分割に強い。過剰なsnapshot依存を避けられる。
- 短所: fixture設計に時間が必要。

### 案C: 完全シミュレーション環境

時刻、天気、PV、CSV、KP-NET、DBをすべて仮想化し、1日または1か月を通しで再生する。

- 長所: 最も強力で、運用シナリオを再現できる。
- 短所: 初期コストが高く、第1段階としては過剰。シミュレータ自体の正しさが課題になる。

## 6. 採用案

**案Bを採用する。** ただし、将来の案Cへ発展できるようfixtureとfake clockを再利用可能に設計する。

### 7.2 backend parity test

同じ正規化入力に対して、各backendが保存する論理値を比較する。DB固有のID、timestamp precision、field orderは比較対象から除外する。

```python
def normalize_cost_rows(rows):
    return [
        {
            "date": row["date"],
            "day_buy_kwh": round(float(row["day_buy_kwh"]), 6),
            "night_buy_kwh": round(float(row["night_buy_kwh"]), 6),
            "cost_yen": round(float(row["cost_yen"]), 2),
        }
        for row in rows
    ]
```

テスト対象は少なくとも以下である。

- 日中・夜間境界の直前/直後
- tier1/tier2境界の直前/一致/直後
- 月締め境界
- 欠損値
- 負値または異常値の扱い
- 同一runの再投入

### 7.3 energy plan characterization

最終payloadを安定化する関数をテスト側に用意する。

除外または固定する項目:

- 生成時刻
- ファイルパス
- 実行環境依存のdiagnostics
- 順序保証のないmapping

厳密比較する項目:

- `target_soc_7_percent`
- `required_night_charge_kwh`
- `predicted_midday_surplus_kwh`
- selected PV method/source
- active constraints
- cost optimizationの選択結果

浮動小数は項目ごとに許容差を決める。すべてを一律丸めしない。

### 7.4 強制充電シナリオ表

最低限、以下をケース化する。

1. 初期SOCが目標以上 → forcedを開始せずstandby。
2. 初期SOCが目標未満 → forced開始。
3. 監視中に目標到達 → forced停止、standby、結果保存。
4. cutoff到達 → forced停止。
5. SOC取得失敗が一時的 → retry。
6. 連続失敗 → fail-safe停止。
7. plan欠損 → 実機操作せずエラー。
8. targetが0または100境界。
9. 再実行時にすでにforced状態。

### 7.5 dashboard契約

- bootstrap payloadは外部JSより前。
- API payloadの必須キーとnull許容を固定。
- JSTの日付境界を固定時刻でテスト。
- empty、1日、月跨ぎ、future periodをテスト。
- Python側集計とJS側表示計算が同一値になる代表ケースを用意する。
