## 1. エグゼクティブサマリー

SolerControlerは、現時点で主要テストが通り、直近のdashboard bootstrap修正も回帰テストで保護されている。一方、保守性上の中心課題は次の4点である。

1. SQLite・PostgreSQL・Firestoreに同一の業務計算が複製されている。
2. `energy_model_main.py`、`cloud_job_runner.py`、`app/dashboard_data.py`、`app/kpnet_workflow.py`、`static/dashboard.js` に責務が集中している。
3. 環境変数、時刻解析、料金計算、CSV探索などの横断処理が複数箇所に存在する。
4. pytestは通るが、mypy、JavaScriptテスト、backend parityがリリースゲートとして統合されていない。

全面改修やフレームワーク置換ではなく、**挙動固定 → 小さな純粋関数の抽出 → backend境界整理 → 状態機械化 → 巨大pipeline分割 → UI/KP-NET分割**の順で進める。この順序は、変更リスクと将来拡張性のバランスが最もよい。

## 4. 目標アーキテクチャ

```text
entrypoints / jobs / HTTP
        |
        v
application services / pipelines
        |
        +------------------+
        |                  |
        v                  v
pure domain logic      effect ports
        |                  |
        |             repositories / KP-NET / HTTP / files
        v                  |
typed models <-------------+
```

原則は次の通りである。

- 計算は純粋関数へ寄せる。
- 環境変数はentrypointで一度だけ解釈する。
- DB固有コードはadapterに閉じ込める。
- 外部I/Oと判断ロジックを分離する。
- backend間で同じ結果になるべき処理はcontract testで固定する。
- 大きな関数は行数だけでなく、入力・出力・副作用の境界で分割する。
- `utils.py` を雑多な共有置き場にしない。時刻、料金、night plan、monitoring CSVなど、ドメイン名で分ける。

## 5. 全体実装案3案

### 案A: 最小限のファイル分割

既存関数をほぼそのまま別ファイルへ移し、importを付け替える。短期間でファイル行数は減るが、業務ロジック重複、環境変数の深い参照、I/Oと判断の混在は残る。

- 長所: 変更量が小さい。短期的に見通しが改善する。
- 短所: 本質的な重複とテスト困難性が残る。
- 適合度: 緊急の可読性改善には有効だが、中長期の基盤として弱い。

### 案B: 段階的なドメイン分離とadapter化

純粋計算、typed model、application service、adapterを段階的に導入する。既存公開関数は当面facadeとして残し、内部を新実装へ委譲する。

- 長所: 正確性、移行安全性、将来拡張性のバランスがよい。
- 短所: 一時的に旧APIと新APIが併存する。characterization testが必須。
- 適合度: 現在の規模、テスト資産、backend数に最も適する。

### 案C: 全面再設計

新パッケージ、新設定基盤、非同期I/O、イベント駆動、TypeScript frontendなどへ一括移行する。

- 長所: 理論上は最も整った構造にできる。
- 短所: 実機制御、DB差、既存運用を同時に壊すリスクが高い。移行期間が長い。
- 適合度: 現段階では過剰。

## 6. 採用案

**案Bを採用する。**

理由は、214件の既存テストを活用しながら、重要なロジックを純粋関数へ移し、旧公開APIを維持したまま段階的に切り替えられるためである。案Aでは重複を残し、案Cでは実機制御とデータ移行リスクが大きすぎる。

## 7. 段階間の依存関係

```text
第1段階 挙動固定
   |
   v
第2段階 共通プリミティブ・設定
   |
   +-----------> 第4段階 dashboard統合
   |
   v
第3段階 DB domain/adapter
   |
   +-----------> 第6段階 energy pipeline
   |
   v
第5段階 強制充電state machine
   |
   v
第7段階 KP-NET / JS分割
```

第4段階は第2段階後に着手できるが、第3段階でnormalized modelが確定するとより安定する。第5段階は第1・第2段階を必須とし、実機副作用を扱うため単独で慎重に進める。第6段階は共通設定とドメイン型が整った後に行う。

## 8. 全体リスク管理

- 各段階は別PRにする。
- 1PRで「移動」「仕様変更」「新機能」を同時に行わない。
- 新旧実装の比較期間を設ける。可能ならshadow calculationを行う。
- DB書込を伴う移行ではdry-runとread-only比較を先に実施する。
- 実機制御変更では必ずstandbyへのfail-safeを維持する。
- 既存環境変数は非推奨期間を設け、いきなり削除しない。
- ロールバックは「旧facadeへ戻す」「feature flagを戻す」「旧書込経路へ戻す」の3層で用意する。
