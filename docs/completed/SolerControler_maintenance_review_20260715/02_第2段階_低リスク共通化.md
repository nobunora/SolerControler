## 文書の使い方

この段階では、実装前に「現在の挙動を壊さないこと」と「次段階の設計を妨げないこと」を同時に満たす必要がある。作業者は、確認済み事実を前提にしてよいが、追加調査項目を推測で埋めてはならない。特に、外部サービス、Firestore、PostgreSQL、KP-NET実機、Cloud Jobのスケジュールに関する事項は、ローカル単体テストだけでは確定できない。

## 1. 目的

巨大な構造変更に入る前に、複数箇所へ散在する小さな共通概念を、ドメイン別の安定した部品へ整理する。対象は環境変数、時刻、料金、monitoring CSV、night plan読み取りなどである。ここではDB orchestratorや充電フローの大きな制御構造は変えない。

## 4. 理想状態と現在のギャップ

| 項目 | 現在 | 理想 | ギャップ |

## 5. 実装案3案

### 案A: `app/utils.py` へ集約

すべての小関数を `utils.py` に移す。

- 長所: import先が一つで簡単。
- 短所: 巨大な雑多モジュールになる。ドメイン境界が失われ、将来再分割が必要。

### 案B: ドメイン別プリミティブ + typed settings

`time_windows.py`, `tariff.py`, `monitoring_csv.py`, `night_plan.py`, `settings/*.py` のように概念単位で分け、既存関数はdeprecation facadeとして委譲する。

- 長所: 責務が明確。第3〜第6段階で再利用できる。
- 短所: ファイル数が増える。命名規則が必要。

### 案C: Pydantic等の設定・schema frameworkを全面導入

すべての環境変数・JSONを外部ライブラリモデルへ置換する。

- 長所: validationとschemaが強い。
- 短所: 依存追加と大規模変更になる。Firestore/CSVの柔軟な欠損データへ適用する設計が必要。

## 6. 採用案

**案Bを採用する。** 現在の依存を増やさず、標準dataclass、Enum、Path、datetimeを中心に進める。schemaが複雑化した時点で限定的にPydantic導入を再評価する。

### 7.1 時刻

```python
@dataclass(frozen=True)
class DailyWindow:
    start: time
    end: time

    def contains(self, value: time) -> bool:
        if self.start <= self.end:
            return self.start <= value < self.end
        return value >= self.start or value < self.end
```

- parseとwindow判定を分離する。
- boundaryは `[start, end)` に統一するか、既存挙動をcharacterization testで固定する。
- HH:MM parserは厳格版とdefault付き設定版を分ける。

### 7.2 料金

optimizerとDB再計算の双方が同じ関数を呼ぶ。丸めは計算途中では行わず、表示・保存境界で明示する。

### 7.3 monitoring CSV

```python
@dataclass(frozen=True)
class MonitoringPoint:
    timestamp: datetime
    load_kwh: float
    pv_kwh: float
    charge_kwh: float
    discharge_kwh: float
    soc_percent: float | None
```

- CSV列名対応を一箇所へ集約。
- SOCは`None`と異常値を区別する。
- `0 <= soc <= 100` とfiniteを検証する。
- latest artifact探索とCSV解析は別関数にする。

### 7.4 night plan

`dict[str, Any]` を直接各所で読むのではなく、最低限のtyped viewを用意する。

```python
@dataclass(frozen=True)
class NightPlanResult:
    target_soc_7_percent: float
    required_night_charge_kwh: float
    predicted_midday_surplus_kwh: float

@dataclass(frozen=True)
class NightPlan:
    forecast_date: date
    sun_hours: float
    temp_c: float
    result: NightPlanResult
```

未知フィールドは保持してもよいが、業務判断に使う値はvalidationを通す。

### 7.5 settings

設定は用途別に分ける。

```python
@dataclass(frozen=True)
class ForcedChargeSettings:
    cutoff: time
    poll_interval_seconds: int
    retry_attempts: int
    retry_delay_seconds: float
```

`from_env()` はentrypointまたはcomposition rootで呼び、domain関数の内部では呼ばない。

## 8. 移行方法

1. 新モジュールへ純粋関数を追加。
2. 既存private関数の本体を新関数呼び出しへ置換。
3. 既存テストと新単体テストを通す。
4. 呼び出し側を順次新APIへ切替。
5. git grepで旧関数参照がなくなったら削除。
6. 旧環境変数名は警告期間を設けてから削除。

一度に全呼び出しを変えない。facadeを使うことでPRを小さくする。

## 9. mypy方針

- `python_version = "3.12"` に合わせる。
- 新規 `app/settings/*`, `time_windows.py`, `tariff.py`, `night_plan.py` はstrict対象にする。
- 既存237件を一括修正せず、typed boundaryを新規モジュールから広げる。
- `Any`を許すadapter入力と、typed domain modelの境界を明示する。

## 10. リスクと対策

- default/clamp差を消してしまう → 既存関数ごとのcharacterization testを先に追加。
- timezone変換が変わる → naive/aware datetimeの扱いを明示しJST fixtureで確認。
- CSVの旧列名を切り捨てる → alias mapを用意。
- night plan欠損時に以前より厳しく落ちる → strict readerとlenient readerを分ける。
