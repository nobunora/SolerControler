## 文書の使い方

この段階では、実装前に「現在の挙動を壊さないこと」と「次段階の設計を妨げないこと」を同時に満たす必要がある。作業者は、確認済み事実を前提にしてよいが、追加調査項目を推測で埋めてはならない。特に、外部サービス、Firestore、PostgreSQL、KP-NET実機、Cloud Jobのスケジュールに関する事項は、ローカル単体テストだけでは確定できない。

## 1. 目的

SQLite、PostgreSQL、Firestoreに複製された業務計算を一つにし、各backendを「読み書きの差」に限定する。今回の全体改善で最も費用対効果が高く、優先度が高い段階である。

## 2. 調査済み状態

### 2.1 重複関数

3backendに次が存在する。

- `ensure_schema`
- `ingest_monitoring_csvs`
- `ingest_settings_summary`
- `ingest_sunshine_from_night_plan`
- `recalc_battery_end_of_day_soc`
- `recalc_battery_pv_charge_end_soc`
- `recalc_cost_daily`
- `recalc_model_hit_rates`
- `record_planned_day_mode`
- `upsert_battery_daily_metrics`
- `upsert_model_parameters_from_plan`

`recalc_cost_daily` は3実装で分岐、ループ、呼び出し構造までほぼ一致する。`ingest_sunshine_from_night_plan` もplan解析と保存が混在している。

### 2.2 依存方向

`postgres_ops.py` と `firestore_ops.py` が `operations_db.py` のprivate関数をimportしている。`operations_db.py` は名称上SQLite adapterだが、実質的に共通domain helperの置き場でもある。このため、SQLite変更が他backendへ影響することがコード構造から分かりにくい。

### 2.3 pipeline分岐

`db_pipeline_main.py` はbackendごとに似た処理列を個別記述している。公開関数名が揃っているため、共通orchestratorへ移行しやすい一方、現在はbackendごとの分岐ブロックが長い。

## 3. 追加調査が必要な箇所

1. 各backendで実際に異なるtransaction境界。
2. Firestore batch上限、分割commit、partial failure時の再実行。
3. PostgreSQL/SQLiteのUPSERT競合キーとunique constraint。
4. timestamp precision、timezone、date型の差。
5. missing field、null、0の意味がbackendごとに違わないか。
6. Firestoreだけに存在するcollection、daily review、GCS連携の位置づけ。
7. どのbackendを正本とし、同期は一方向か双方向か。
8. 同一run keyのidempotencyと再処理要件。
9. migrationを必要とするschema変更の有無。

## 4. 理想状態と現在のギャップ

| 観点 | 現在 | 理想 | ギャップ |
|---|---|---|---|
| domain計算 | adapter内に複製 | backend非依存の単一実装 | driftリスクが高い |
| adapter | 計算とI/Oが混在 | fetch/saveのみ | テストに実DBが必要 |
| pipeline | backend別処理列 | 共通service + repository | 分岐が長い |
| 型 | dict/DB row中心 | typed input/output model | backend差を比較しにくい |
| private helper | SQLiteから他backendがimport | domain moduleへ正方向依存 | 名称と責務が不一致 |

## 5. 実装案3案

### 案A: 共通helperだけ切り出す

private helperを `operations_common.py` へ移し、3backend関数はそのまま残す。

- 長所: 変更が小さい。
- 短所: `recalc_cost_daily` 等の大きな処理列は3重のまま。長期的driftを止めきれない。

### 案B: 共通domain service + repository adapter

計算をpure domainへ、DB読み書きをrepositoryへ、処理順をapplication serviceへ分離する。既存関数はfacadeとして新serviceへ委譲する。

- 長所: 重複を本質的に解消し、parity testが容易。段階移行可能。
- 短所: modelとrepository契約の設計が必要。

### 案C: 単一正本DB + 他backendを同期先に限定

例えばPostgreSQLを正本とし、Firestore/SQLiteはexport・cacheにする。

- 長所: 書込経路が単純になる。
- 短所: 運用要件の変更が大きい。オフライン/クラウド用途を損なう可能性。

## 6. 採用案

**案Bを採用する。** 正本DBの整理は追加調査後に別意思決定とする。現時点では3backendを維持しつつ、domain計算とpipelineを共通化する。

### 7.1 domain model

```python
@dataclass(frozen=True)
class EnergyInterval:
    timestamp: datetime
    bought_kwh: float
    sold_kwh: float
    pv_kwh: float
    load_kwh: float

@dataclass(frozen=True)
class DailyCostResult:
    date: date
    day_buy_kwh: float
    night_buy_kwh: float
    total_cost_yen: float
    tariff_mode: str
```

DB rowをそのままdomainへ渡さず、adapterで型変換する。

### 7.2 repository protocol

```python
class OperationsRepository(Protocol):
    def pipeline_run_exists(self, run_key: str) -> bool: ...
    def load_energy_intervals(self, period: DateRange) -> Sequence[EnergyInterval]: ...
    def save_daily_costs(self, rows: Sequence[DailyCostResult]) -> None: ...
    def load_settings_summary(self, path: Path) -> SettingsSummary: ...
    def save_battery_metrics(self, metrics: Sequence[BatteryDailyMetrics]) -> None: ...
```

repositoryを過度に巨大化しない。読み書き用途ごとに小さなProtocolへ分割してもよい。

### 7.3 application service

```python
class OperationsPipelineService:
    def __init__(self, repo: OperationsRepository, tariff: TariffSettings):
        self.repo = repo
        self.tariff = tariff

    def run(self, request: PipelineRequest) -> PipelineResult:
        ...
```

処理順は一度だけ定義する。backend選択はentrypointで行う。

### 7.4 adapter

adapterは以下だけを担当する。

- 接続生成
- schema/collection準備
- query
- row/document変換
- transaction/batch
- backend固有の再試行

料金計算、SOC計算、plan解析を持たせない。

## 8. 移行手順

### Step 1: 料金計算

1. 既存3関数の入力と出力をfixtureで固定。
2. `app/operations/domain/cost.py` にpure関数を作る。
3. 各既存 `recalc_cost_daily` が、DBからintervalを読み、共通関数を呼び、保存する形へ変更。
4. backend parityを確認。

最初の対象として最適である。重複約520行の大部分を削減できる。

### Step 2: night plan解析

`ingest_sunshine_from_night_plan` からplan解析を分離する。adapterは `NightPlanMetrics` を保存するだけにする。

### Step 3: battery metrics

summaryとplanからmetricsを作る関数を共通化する。

### Step 4: model hit ratesとSOC再計算

入力rowをnormalized modelへ変換し、計算をdomainへ移す。

### Step 5: pipeline統合

`db_pipeline_main.py` のbackend別処理列を `OperationsPipelineService` へ置換する。既存entrypointと環境変数は維持する。

### Step 6: 旧facade整理

外部から参照される関数を確認し、compatibility facadeを残すか削除する。

## 9. backend差の扱い

- Firestoreのbatch分割はadapter内部。
- PostgreSQL/SQLiteのtransactionはUnit of Workまたはcontext managerでadapter内部。
- Firestore固有daily reviewは、共通domainに必要な機能か、dashboard read model専用かを明示する。
- GCS uploadはoperations domainではなくarchive portとして分離する。
- schema migrationはadapterごとに管理する。

## 11. ロールバック

- 既存公開関数をfacadeとして残す。
- `OPERATIONS_PIPELINE_V2` のfeature flagで新旧切替を可能にする。
- 初期は新計算結果を保存せずログ比較するshadow modeを用意する。
- 差異が許容範囲を超えた場合は旧経路へ戻す。

## 12. 完了条件

- 主要domain計算が3backendで単一実装になる。
- `postgres_ops.py` / `firestore_ops.py` が `operations_db.py` のprivate helperをimportしない。

- 同一run再実行の結果がidempotentである。
